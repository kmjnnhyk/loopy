loopy.js 3개의 층 예제: bell-agent의 일 ① 지금(손코딩) ② LangChain ③ loopy 개념 매핑 정리
Design Note · 왜 새 언어인가

bell-agent를 짜는 세 가지 방법
손코딩 vs LangChain vs loopy

같은 일 — "디자이너의 자연어 요청 → 코드 수정(tool 호출 루프) → 빌드 검증 → 사람 승인 → PR 생성" — 을 세 방식으로 짜 보면, LangChain이 무엇이고, bell-agent가 그것과 어떤 관계이며, loopy가 어디서 더 쉬워지는지가 한눈에 보입니다.

A먼저: "LangChain이 뭐냐"를 3개의 층으로

바닐라 JS로도 웹을 만들지만 큰 앱은 관리가 안 돼 React를 쓰듯, LLM 앱에도 같은 3개 층이 있습니다.

맨 아래 · 모델 SDK
API 한 번 호출
웹 비유: fetch로 서버 때리기
@anthropic-ai/sdk → bell-agent가 직접 씀 ✓
가운데 · 프레임워크
에이전트 루프 · 툴 · 상태 · 구조
웹 비유: React
LangChain / LangGraph / loopy → bell-agent엔 없음 → 손으로 재구현
맨 위 · 애플리케이션
실제 제품
웹 비유: 실제 웹사이트
Slack/Jira 봇 = bell-agent ✓
핵심 답: "bell-agent/apps/api의 역할이 LangChain과 같냐?" → 앱 전체로는 다름(앱 vs 라이브러리). 하지만 bell-agent 안에 박힌 core/ + 손으로 짠 에이전트 루프 + 툴 정의는 LangChain이 하는 일과 정확히 같은 역할입니다. 즉 bell-agent = "LangChain 없이, LangChain이 할 일을 인라인으로 손수 짠 앱". loopy는 바로 그 가운데 층(LangChain 슬롯)에 들어갑니다.

B예제: bell-agent의 "디자인 → PR" 에이전트

아래 흐름 하나를 세 방식으로 짜 봅니다. tool 호출 루프 · 구조화 출력 · 분기 · 사이클(빌드 실패 시 재시도) · 사람 승인(일시정지) · 영속성을 모두 포함하는 현실적인 예제입니다.

①지금 — 손코딩 (bell-agent의 실제 코드)

프레임워크 없이 Anthropic SDK 위에 모든 걸 직접. 아래는 apps/api/src/agents/design/code-generator.js의 실제 코드(축약).

core/claude.js + code-generator.js순수 JS · 타입 없음
// 1) 모델 래퍼 — 손으로 짠 "ChatAnthropic"
async function callSonnet(systemPrompt, messages, tools) {
  return client.messages.create({ model: "claude-sonnet-4", system: systemPrompt, messages, tools });
}

// 2) 툴 정의 — JSON Schema를 손으로 (타입 추론 0)
const TOOLS = [
  { name: "edit_file",   input_schema: { type:"object", properties:{ file_path:{type:"string"}, old_string:{type:"string"}, new_string:{type:"string"} }, required:["file_path","old_string","new_string"] } },
  { name: "create_file", input_schema: { /* ... 또 손으로 ... */ } },
  { name: "read_file",   input_schema: { /* ... 또 손으로 ... */ } },
];

// 3) 에이전트 루프 — 손으로 짠 AgentExecutor (195~264줄)
let iterations = 0;
while (iterations <= MAX_READBACKS) {                  // ← 루프·종료조건 직접
  const response = await callSonnet(SYSTEM, messages, TOOLS);
  const parsed  = parseToolCalls(response);            // ← tool_use 블록 직접 파싱
  if (parsed.readRequests.length===0 && parsed.changes.length===0) break;
  messages.push({ role:"assistant", content: response.content }); // ← 히스토리 누적 직접 (= spec 버그 #2)
  const toolResults = [];
  for (const change of parsed.changes) {
    const result = await applyChanges([change], repoPath);   // 부수효과를 루프 한복판에서
    toolResults.push({ type:"tool_result", tool_use_id: change.toolUseId, content: result.applied.length? "적용 완료" : "적용 실패..." });
  }
  messages.push({ role:"user", content: toolResults });   // ← tool_result 피드백 직접
  iterations++;
}

// 4) 구조화 출력 — regex + JSON.parse + 조용한 폴백 (parser.js / classifier.js)
const name = (data.content?.[0]?.text || "").trim();
if (name && name.startsWith("design/")) return name;
return `design/request-${Date.now()}-${slug}`;   // ← 실패를 성공처럼 폴백 + 비결정성
무엇이 아픈가
  • 루프를 매번 손으로 → 히스토리 누적 한 줄 빠뜨리면 "중복 편집" 버그(spec #2). 모든 에이전트가 이 루프를 또 짬.
  • 툴이 JSON Schema 손코딩 → 입력 타입이 코드와 분리, block.input.file_path는 any.
  • 구조화 출력 = regex/JSON.parse + 조용한 폴백 → 파싱 실패가 "성공"으로 위장(failOpenVerdict, 브랜치명 폴백).
  • 상태는 가변 ctx bag, 진행상황은 console.log+WS 수동, 영속성은 DB 상태 문자열 — 전부 흩어짐.
  • "어디에 뭘 두나" 규칙 없음 → 같은 일(LLM이 코드 편집)을 한 곳은 이 루프로, 다른 곳은 spawn("claude")로 두 번 구현.

②LangChain / LangGraph로

프레임워크를 쓰면 루프·파싱·체크포인트가 공짜가 됩니다. bell-agent의 손코딩 절반이 사라지죠. 하지만 — 타입 추론이 군데군데 끊기고, "어디에 뭘 두나"는 여전히 당신 몫입니다.

graph.ts (한 파일에 다 — 폴더 규칙은 LangChain이 안 정해줌)@langchain/langgraph
import { ChatAnthropic } from "@langchain/anthropic";
import { tool } from "@langchain/core/tools";
import { StateGraph, Annotation, START, END } from "@langchain/langgraph";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { z } from "zod";

// 툴 — zod 스키마로 입력은 추론됨 (좋음). 하지만…
const editFile = tool(
  async ({ path, oldString, newString }) => {
    return JSON.stringify(await repo.applyEdit(path, oldString, newString)); // ← 출력은 문자열로 (타입 유실)
  },
  { name:"edit_file", schema: z.object({ path:z.string(), oldString:z.string(), newString:z.string() }) } // repo는 클로저로 — 주입 안 됨, 타입에 안 보임
);

// 코드 에이전트 — 루프는 공짜! (createReactAgent가 소유) ✓
const coder = createReactAgent({ llm:new ChatAnthropic({ model:"claude-sonnet-4" }), tools:[editFile, createFile, readFile] });
// …단, createReactAgent는 자유 텍스트 반환 → 구조화 결과는 따로 파싱해야 함

// 분류기 — withStructuredOutput로 regex/JSON.parse 추방 ✓
const classifier = new ChatAnthropic({ model:"claude-haiku-4-5" })
  .withStructuredOutput(z.object({ kind:z.enum(["talk","code_change","question"]), reason:z.string() }));

// 상태 — 채널마다 reducer 선언 (좋은 아이디어! loopy도 이걸 훔침) ✓ …근데 타입을 또 손으로
const State = Annotation.Root({
  kind:     Annotation<string>(),                                  // ← enum 아님, 그냥 string 으로 퇴화
  changes:  Annotation<Change[]>({ reducer:(a,b)=>a.concat(b), default:()=>[] }),
  build:    Annotation<BuildResult>(),
  attempts: Annotation<number>({ reducer:(a,b)=>a+b, default:()=>0 }),
});

const graph = new StateGraph(State)
  .addNode("classify", async(s)=>({ kind:(await classifier.invoke(s.request)).kind }))   // 노드 이름이 문자열
  .addNode("code",  async(s)=>({ changes: parseChanges(await coder.invoke({messages:[...] })) }))
  .addNode("build", async(s)=>({ build:await buildRepo(s), attempts:1 }))
  .addNode("pr",    async(s)=>({ /* ... */ }))
  .addEdge(START, "classify")
  .addConditionalEdges("classify", (s)=> s.kind==="code_change" ? "code" : END)         // ← 문자열 반환, 오타 잡아줄 타입 없음
  .addEdge("code", "build")
  .addConditionalEdges("build", (s)=> s.build.ok ? "pr" : s.attempts<3 ? "code" : END)
  .addEdge("pr", END)
  .compile({ checkpointer:new SqliteSaver(...) });   // 영속성·resume 있음 ✓ (좋음)

const out = await graph.invoke({ request }, { configurable:{ thread_id:"t-1" } });
LangChain이 해결한 것 (손코딩 대비)
  • 에이전트 루프가 공짜 (createReactAgent) — 버그 #2 소멸
  • 구조화 출력 withStructuredOutput — regex/JSON.parse 추방
  • 채널별 reducer 상태 + 체크포인트/resume 내장
  • 툴 입력은 zod로 추론
그래도 남는 불편 (= loopy가 노리는 틈)
  • 노드가 문자열("code") — 오타=런타임 에러, 자동완성·리팩토링 약함
  • 툴 deps가 클로저(repo) — 주입·테스트·타입에 안 나타남
  • 툴 출력 타입 유실(문자열 반환) → 다시 파싱
  • State 타입을 손으로 또 적음(Annotation<string>) — 스키마에서 추론 안 됨
  • HITL(사람 승인)은 interrupt가 있지만 결선이 번거로움
  • ★ 폴더 구조·툴 레지스트리·"어디에 뭘 두나" 규칙이 전혀 없음 — 당신이 처음 말한 바로 그 문제

③loopy로 — 같은 일, 규격화 + 타입세이프

loopy는 LangChain의 좋은 부분(공짜 루프·reducer 채널·체크포인트)을 가져오되, "한 파일 한 단위 · 규정된 폴더 · 값 참조 · 끝까지 추론 · HITL을 한 줄로"를 강제합니다. React가 컴포넌트에 한 것처럼.

① 규정된 폴더 — "어디에 뭘 두나"에 단 하나의 답

② tool — 자기 파일에서 자기를 선언 (스키마+로직+deps 한곳)

src/tools/editFile.tool.tsloopy
import { tool } from "loopy";
import { z } from "zod";

export default tool({
  name: "edit_file",
  description: "기존 파일의 특정 부분을 수정한다.",
  input:  z.object({ path:z.string(), oldString:z.string(), newString:z.string() }),
  output: z.object({ applied:z.boolean(), reason:z.string().optional() }),  // 출력도 스키마 → 타입 보존
  run: async ({ path, oldString, newString }, ctx) => {                     // 입력 전부 타입드
    return ctx.deps.repo.applyEdit(path, oldString, newString);             // ctx.deps.repo 주입됨·타입드
  },
});
// 타입: Tool<"edit_file", {path,oldString,newString}, {applied,reason?}, { repo: GitRepo }>
//   name=리터럴, 입출력=스키마 추론, deps=사용처에서 추론 → 레지스트리가 합산

③ agent — 손으로 짠 195줄 루프가 이 선언으로

import { agent, stepCountIs } from "loopy";
import editFile   from "../tools/editFile.tool";
import createFile from "../tools/createFile.tool";
import readFile   from "../tools/readFile.tool";

export default agent({
  name: "coder",
  model: "claude-sonnet-4-6",
  instructions: WIZKEY_SYSTEM_PROMPT,
  tools: [editFile, createFile, readFile],     // ← 값으로 import = LLM manifest 단일 소스, go-to-def
  output: z.object({ appliedChanges:z.array(z.object({ path:z.string() })), summary:z.string() }),

});
// 루프·히스토리 누적·tool_result 피드백·재시도·파싱 = 전부 런타임 소유. 당신은 '선언'만.

⑤ workflow — 노드는 값, 라우터·사이클·HITL 전부 타입드

src/workflows/shipDesign.workflow.tsloopy
import { workflow, channel, append, last, END } from "loopy";
import classifier from "../agents/classifier.agent";
import coder      from "../agents/coder.agent";
import buildRepo  from "../tools/buildRepo.tool";
import openPr     from "../tools/openPr.tool";

export default workflow({
  name: "ship-design",
  input: z.object({ request:z.string(), repo:z.string() }),
  state: {                                              // 채널마다 reducer (LangGraph서 훔침) — 단, 타입은 추론
    changes:  channel(append).initial([]),
    build:    channel(last),
    attempts: channel((a,b)=>a+b).initial(0),
  },
})
  .step("classify", classifier)                          // 노드 = 값(agent). 오타날 문자열 없음
  .step("code",     coder)
  .step("build",    buildRepo)
  .step("approve",  approveGate())                       // 사람 승인 (아래)
  .step("pr",       openPr)
  .start("classify")
  .branch("classify", (s) => s.classify.kind==="code_change" ? "code" : END)  // s.classify.kind 는 enum, 자동완성됨
  .edge("code", "build")
  .branch("build", (s) => s.build.ok ? "approve" : s.attempts<3 ? "code" : END) // 사이클 — 타입 체크됨
  .edge("approve", "pr");

⑥ HITL — bell-agent의 awaiting_base DB 상태 머신이 한 줄로

src/workflows/approveGate.tsloopy
import { step } from "loopy";

export const approveGate = () => step("approve", async (s, ctx) => {
  const ok = await ctx.interrupt({ ask:"이 변경으로 PR 만들까요?", preview:s.changes }); // 일시정지 + 체크포인트
  if (!ok) return ctx.end("사용자가 취소함");
  return {};
});
// 실행이 여기서 멈추고 상태가 store에 저장됨. 며칠 뒤 사람이 Slack에서 승인 누르면:
//   await runtime.resume(threadId, { value: true });  → 멈춘 지점부터 재개

⑦ 중앙 레지스트리 + 실행

import { defineLoopy } from "loopy";
import { claude } from "loopy/models";
import { sqliteStore } from "loopy/store";
import shipDesign from "./src/workflows/shipDesign.workflow";

export default defineLoopy({
  models: { "claude-sonnet-4-6": claude(), "claude-haiku-4-5": claude() },
  deps:   { repo: new GitRepo() },     // 모든 tool의 deps 합집합을 만족해야 함 → 컴파일 체크
  store:  sqliteStore("./runs.db"),     // 영속성/resume
  workflows: { shipDesign },
});

// ── 실행 (Slack 핸들러 등 어디서든) ──────────────────
const run = await runtime.run("shipDesign", { request, repo }, { threadId: slackThreadTs });
// 진행상황은 step 이벤트로 자동 방출 → Slack/WS/로그 sink에 그대로 연결 (수동 emit 끝)
// 사람이 승인하면:  await runtime.resume(slackThreadTs, { value:true });
loopy가 강제한 것 (= 당신이 원한 "규격화")
  • 한 파일 = 한 단위, *.tool.ts/*.agent.ts 네이밍 — 어디에 뭘 두는지 단 하나의 답
  • tool은 값으로 import → 어떤 agent가 무슨 tool을 부르는지 타입·go-to-def로 추적, 안 쓰는 tool 감지
  • 입력·출력·deps 전부 추론 → any 없음, Annotation<string>처럼 타입 다시 적기 없음
  • 노드는 문자열이 아니라 값/타입드 키 → 오타가 컴파일 에러
  • 구조화 출력·루프·진행상황·영속성·HITL이 전부 1급 → 손코딩 0

D한 줄 정리

LangChain

가운데 프레임워크 층을 채워, bell-agent가 손으로 짠 루프·파싱·체크포인트를 공짜로 만들어 줍니다. 손코딩보다 확실히 낫습니다. 하지만 타입 추론이 끊기고, 무엇보다 당신이 처음 말한 "어디에 뭘 두고 어떻게 관리하나"(폴더·레지스트리·규칙)에 답이 없습니다.

loopy

같은 프레임워크 층을, "타입세이프 + 구조 강제 + 저-보일러플레이트"라는 당신 취향으로 다시 채웁니다. React가 컴포넌트로 한 일을 에이전트/툴/워크플로우에 합니다 — 한 파일 한 단위, 규정된 폴더, 값 참조, 끝까지 추론, 루프·영속성·HITL은 선언 한 줄.

그래서 loopy ≠ LangChain 래퍼. loopy는 LangChain과 같은 슬롯의 경쟁 프레임워크이되, 빠진 조각(구조 표준화·끝까지 타입 추론)을 메우는 것이 존재 이유입니다. bell-agent를 loopy로 다시 쓰면 core/claude.js · parser.js · code-generator.js의 루프가 통째로 사라지고, 선언만 남습니다.
다음 단계 — 이 그림이 맞으면, 막혔던 두 질문을 이 맥락에서 다시 풉니다:
  • 레지스트리를 얼마나 자동으로 조립할까 (명시적 중앙 vs Next.js식 파일기반 생성) — "표준화 강도"
  • agent가 tool을 참조하는 방식 (값 import vs 이름 키)
그 다음 ③ State·이벤트로그·영속성 내부 와 ④ 타입세이프티 기계장치(end-to-end 추론·Schema-Aligned Parsing)로 들어갑니다.
