# team, 쉽게 이해하기 — 리뷰어를 위한 안내

> 이 문서는 **개념 이해용**입니다. 실제 구현에 쓰는 정밀 스펙은 `docs/superpowers/specs/2026-06-30-team-design.md`에 따로 있습니다. 여기서는 타입·에지케이스를 빼고, 무엇을 왜 하는지 사람 말로만 설명합니다.

읽는 순서를 일부러 이렇게 잡았습니다: **채널 → ReviewResult → passTo vs router → 나머지 코드 → 전체 흐름.** 앞 개념 위에 뒤 개념이 얹히기 때문입니다.

---

## 0. 한 문장 + 사무실 비유

**team = 직원(agent) 여러 명이 하나의 사건을 이어받아 처리하는 팀.**

이 문서가 계속 쓰는 예제는 **PR/이슈 트리아지 팀**입니다. 사무실로 비유하면:

```
이슈 접수 → [접수 담당] → [버그 담당] 또는 [문서 담당] → [검토 담당] ─┬─ 승인 → 끝
                                                                     └─ 반려 → 담당자에게 되돌림
```

- **접수 담당(triage)**: 이슈를 읽고 "이건 버그네 / 이건 문서 요청이네" 판단해서 담당자에게 넘김
- **버그 담당(bugFixer) / 문서 담당(docsWriter)**: 실제로 일 처리, 끝나면 검토 담당에게 넘김
- **검토 담당(reviewer)**: 결과 확인 → OK면 끝, 미흡하면 담당자에게 되돌림

사람 팀이 일하는 방식 그대로입니다. 이제 이걸 코드로 어떻게 표현하는지, 낯선 단어부터 하나씩 풉니다.

---

## 1. 채널이 뭔가 (가장 먼저)

**채널 = 팀이 다 같이 보는 화이트보드의 "칸" 하나.**

팀이 협업하려면 정보를 어딘가 공유해야 합니다. 그 공유 공간이 화이트보드이고, 화이트보드는 여러 **칸**으로 나뉩니다. 각 칸이 "채널" 하나입니다. 칸마다:
- **이름표**가 있고 (예: `issue`, `review`),
- **새 값이 오면 기존 값과 어떻게 합칠지 규칙**이 붙어 있습니다.

그 "합치는 규칙" 때문에 채널 종류가 나뉩니다:

| 종류 | 뜻 | 비유 | 예 |
|---|---|---|---|
| `lastChannel` | **마지막 값만 유지** (새 값이 오면 덮어씀) | 화이트보드에 지우고 다시 씀 | `review`(검토 결과) |
| `listChannel` | **계속 쌓음** (새 값이 오면 뒤에 붙임) | 회의록에 줄줄이 적음 | `transcript`(대화 기록) |
| `inputChannel` | **시작할 때 밖에서 넣어주는 입력** | 사건 접수 서류 | `issue`(처리할 이슈) |

**"그냥 변수 쓰면 안 되나?"** — 안 됩니다. loopy는 모든 걸 이벤트 로그에 기록해서 **나중에 그대로 재생(replay)하고 타임트래블 디버깅**을 합니다. 채널은 그 "기록되고 되감을 수 있는 변수"입니다. (이게 loopy 전체를 관통하는 `state = fold(reducer, log)` 아이디어 — 지금 깊이 안 봐도 됩니다.)

**team이 자동으로 깔아주는 칸 2개** (당신이 선언 안 해도 생김):
- **`transcript`**: 지금까지 오간 대화 전부 (누가 뭐라고 했는지). 늦게 온 검토 담당도 이 칸 보면 앞 사람들이 뭘 했는지 앎.
- **`nextAgent`**: "다음 차례는 누구" 적어두는 **쪽지 한 장**. (뒤에서 핵심이 됩니다.)

당신이 직접 선언하는 칸은 당신 일에 필요한 것만: 이 예제에선 `issue`(입력)와 `review`(검토 결과).

## 2. ReviewResult가 뭔가

**`ReviewResult`는 loopy 개념이 아닙니다. 이 트리아지 예제가 만든 "검토 결과" 데이터일 뿐입니다.**

검토 담당(reviewer)이 일을 끝내면 결론을 냅니다. 그 결론의 모양이 `ReviewResult`입니다. 두 경우뿐이죠:

```ts
type ReviewResult =
  | { approved: true;  notes: string }                                    // 승인: 통과 + 메모
  | { approved: false; assignee: "bugFixer" | "docsWriter"; notes: string } // 반려: 누구한테 되돌릴지 + 메모
```

- **승인**이면 `approved: true` + 메모.
- **반려**면 `approved: false` + **누구한테 되돌릴지**(`assignee`) + 메모.

**왜 이렇게 둘로 갈라 썼나(discriminated union):** "반려일 때는 반드시 담당자를 지정"하게 **타입으로 강제**하려고요. 승인엔 담당자가 필요 없고, 반려엔 필수입니다. 이렇게 해두면 "반려인데 담당자를 안 정하는" 실수가 컴파일 단계에서 막힙니다.

**핵심:** ReviewResult는 그냥 이 예제의 도메인 데이터입니다. 당신 앱이 주문 처리면 여기 `OrderResult`가, 분석이면 `AnalysisReport`가 들어갑니다. loopy가 강요하는 게 아니라 **당신이 정의하는 결과 타입**이에요.

## 3. passTo vs router — 이게 핵심 (당신이 헷갈린 부분)

둘 다 **"다음 차례는 누구?"를 정하는 것**입니다. 차이는 딱 하나: **누가 정하느냐.**

### passTo = 직원(AI)이 스스로 넘긴다
접수 담당한테 미리 "넌 버그담당이나 문서담당한테 넘길 수 있어"라고 알려주면(`passTo: ["bugFixer", "docsWriter"]`), 접수 담당 AI가 이슈를 읽고 **스스로 판단해서** "버그담당한테 넘기기" 버튼을 누릅니다.

- "이게 버그냐 문서냐"는 **이슈를 실제로 읽어봐야 아는 판단**이라, AI만 할 수 있습니다.
- AI가 `pass_to_bugFixer()` 버튼을 누르면 → 아까 그 **`nextAgent` 쪽지에 "bugFixer"라고 적힙니다.**

### router = 코드가 규칙으로 정한다
"검토가 승인되면 끝낸다", "반려면 담당자한테 되돌린다" — 이건 **AI 판단이 아니라 딱 정해진 규칙**이라 코드가 처리합니다. 이게 `.router(...)`입니다.

### 그래서 당신이 헷갈렸던 코드 블록을 한 줄씩

```ts
.router((s) => {                          // s = 지금 화이트보드 전체 상태 (모든 칸의 현재 값)
  if (s.nextAgent) return s.nextAgent;    // ① AI가 넘긴 쪽지가 있으면, 그 사람 차례
  if (s.review?.approved) return END;    // ② 검토가 "승인"이면, 끝
  if (s.review) return s.review.assignee; // ③ 검토가 "반려"면, 지정된 담당자 차례
  return END;                             // ④ 넘길 쪽지도, 검토 결과도 없으면, 끝
})
```

한 줄씩 사람 말로:
- **①** `s.nextAgent`는 아까 그 "다음 차례 쪽지"입니다. **AI가 `pass_to_*` 버튼으로 적어둔 게 있으면 그걸 최우선으로 따릅니다.** (참고: 맨 처음 turn에는 이 쪽지가 `entry`, 즉 접수 담당으로 미리 세팅돼 있어서 접수 담당이 첫 타자가 됩니다.)
- **②** 검토 담당이 "승인" 결과를 냈으면(`review.approved`) → **끝**(`END`).
- **③** 검토가 "반려"면 → 검토 담당이 지정한 담당자(`assignee`)에게 **되돌림.**
- **④** 넘길 쪽지도 없고 검토 결과도 없으면 → **끝.**

### 왜 쪽지(nextAgent)를 검토결과(review)보다 먼저 보나?
반려로 되돌아간 버그 담당이 일을 다시 하고 **검토 담당한테 다시 넘길 때** 새 쪽지(`nextAgent="reviewer"`)를 적습니다. 이때 *낡은 반려 결과*보다 *새 쪽지*를 먼저 봐야 검토 담당한테 제대로 돌아갑니다. 안 그러면 낡은 "반려" 결과만 보고 버그 담당을 무한 반복하게 됩니다. (이게 검증 과정에서 잡은 실제 버그였습니다.)

### 한눈 정리

| | passTo | router |
|---|---|---|
| **누가 정하나** | 직원(AI)이 스스로 | 코드가 규칙으로 |
| **언제 쓰나** | "버그냐 문서냐"처럼 **읽어봐야 아는 판단** | "승인되면 끝"처럼 **정해진 규칙** |
| **어떻게** | AI가 `pass_to_X()` 버튼 → 쪽지에 적힘 | `.router()`가 쪽지·결과 보고 결정 |
| **필수?** | 선택 | 선택 (생략하면 "쪽지 따라가, 없으면 끝"이 기본) |

**즉 둘은 경쟁 관계가 아닙니다.** router가 최종 결정권자이고, passTo는 AI가 "저 사람한테 넘겨주세요"라고 **쪽지로 요청**하는 것입니다. 단순한 팀은 passTo만으로 돌아가고(router 생략), "규칙 기반 종료·되돌림"이 필요할 때만 router를 씁니다.

## 4. 나머지 코드 (`.writes` / `defineLoopy` / `rt.run`)

```ts
.writes({ reviewer: "review" })
```
검토 담당이 낸 결과(`ReviewResult`)를 **어느 칸에 적을지** 알려주는 매핑입니다. "reviewer의 결과물을 `review` 칸에 써라." 이게 있어야 위 router의 `s.review`가 채워집니다.

```ts
const rt = defineLoopy({ teams: { prTriage } }).provide(/* deps */);
```
- `defineLoopy(...)` = **앱 등록소.** 만든 팀·에이전트를 여기 등록합니다.
- `.provide(...)` = **실제 의존성 주입.** (에이전트들이 쓰는 실제 도구·API 등을 여기서 꽂아줍니다.)

```ts
const out: ReviewResult | null = await rt.run("prTriage", { issue }, { threadId: "t-7" });
```
- `rt.run("prTriage", { issue })` = **"이 이슈로 트리아지 팀을 실행"**. `{ issue }`가 아까 `inputChannel`(입력 칸)에 들어갑니다.
- `threadId: "t-7"` = 이 실행의 **식별표**. 나중에 중단됐다가 재개(resume)하거나 재생할 때 씁니다.
- 반환 `out: ReviewResult | null` = **`review` 칸의 최종 값.** 검토 담당이 결론을 냈으면 `ReviewResult`, 아무도 검토를 안 하고 끝났으면 `null`입니다. (안전장치 `maxTurns`에 걸려 멈추면 값을 돌려주는 게 아니라 에러를 던집니다.)

## 5. 전체 흐름 한 번 (한 사건이 도는 모습)

버그 이슈가 들어와서, 한 번 반려됐다가 통과되는 경우:

| 턴 | 활성 직원 | 하는 일 | "다음 차례 쪽지"·"검토결과" | router 결정 |
|---|---|---|---|---|
| 0 | **접수 담당** | 이슈 읽음 → "버그다" → 버그담당에게 넘김 | 쪽지 = bugFixer | → 버그 담당 |
| 1 | **버그 담당** | 코드 고침 → 검토 담당에게 넘김 | 쪽지 = reviewer | → 검토 담당 |
| 2 | **검토 담당** | 검토 → "미흡, 버그담당 반려" | 결과 = 반려(assignee=bugFixer) | → 버그 담당 (되돌림) |
| 3 | **버그 담당** | 다시 고침 → 검토 담당에게 넘김 | 쪽지 = reviewer | → 검토 담당 |
| 4 | **검토 담당** | 검토 → "승인!" | 결과 = 승인 | → **끝(END)** |

최종 반환 = 승인된 `ReviewResult`. 그리고 이 모든 과정이 이벤트 로그에 기록돼서, 똑같이 재생하거나 중간에 사람이 개입(HITL)할 수 있습니다.

## 6. 한 장 요약

- **채널** = 팀 공유 화이트보드의 칸 (기록·재생되는 변수). `last`(덮어쓰기)/`list`(쌓기)/`input`(입력).
- **ReviewResult** = 이 예제의 "검토 결과" 데이터 (loopy 것 아님, 당신 도메인 타입).
- **passTo** = AI가 스스로 "다음은 저 사람" 하고 넘기는 것 (읽어봐야 아는 판단).
- **router** = 코드가 규칙으로 "다음은 누구/끝" 정하는 것 (정해진 규칙).
- 둘은 협력: router가 결정하고, passTo는 쪽지로 요청.

궁금한 지점이 더 있으면 말씀해 주세요. 이해가 맞춰지면 이 개념들을 바탕으로 구현 계획(writing-plans)으로 넘어가겠습니다.
